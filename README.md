# RVR Analysis Project

## 1.项目背景

机场 RVR（Runway Visual Range）设备存在间歇性数据丢失的问题。

本项目旨在利用 Python、Pandas、SQL 对历史数据进行分析，
定位故障原因，并逐步实现自动化故障诊断工具。

## 2.数据审视

### 审视内容：

- 文件路径检查
- 文件基本信息获取
- 文件预览（前10行）
- 分隔符识别

### 遇到的问题及解决方案


| 模块         | 遇到的问题                                                              | 思考总结                                                             |
| ------------ | ----------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 文件编码识别 | 为什么检测编码不能直接使用 `open(..., "r")`？                           | 编码未知时，应先使用二进制方式（`rb`）读取原始字节，再进行编码检测。 |
| 文件编码识别 | 为什么 `chardet` 返回的是推测而不是确定结果？                           | 编码检测基于概率，`confidence` 表示可信度，需要结合实际文件验证。    |
| 文件预览     | 为什么使用 `repr()` 而不是 `print()`？                                  | `repr()` 可以显示隐藏字符（如 `\n`、`\t`），便于分析文件真实结构。   |
| 列名检测     | 为什么 `return` 放在 `if` 中还能结束 `for`？                            | `return` 结束的是整个函数，而不仅仅是 `if` 或 `for`。                |
| 列名检测     | 为什么 `print("未识别到列名")` 不会执行？                               | 找到列名后函数已 `return`，后续代码不会继续执行。                    |
| 分隔符检测   | 为什么不能写死 `split("\t")`？                                          | 程序不应依赖人为假设，应先自动检测分隔符，再传递给后续函数。         |
| 参数传递     | 为什么 `extract_columns(lines, separator, header_index)` 需要三个参数？ | 函数应通过参数获取所需数据，而不是依赖外部变量，提高复用性。         |
| 函数设计     | 为什么 `separator` 没定义却没有报错？                                   | 形参在函数定义阶段只是占位符，真正调用函数时才会接收实际值。         |

### 技术收获

- 构建`文件编码检测`函数时，利用二进制方式(`rb`)读取原始字节，使用工具：`chardet.detect(raw_data)`。
- 利用 enumerate(start=1) 获取文件真实行号，检测到目标后立即 return，避免无效扫描。
- 最后调用函数工具时，检查是否有重复调用的情况。


设计思路

`inspect_data.py` 的职责不是读取数据，而是识别文件结构。

因此，它负责：

- 文件路径检查
- 文件基本信息
- 文件编码识别
- 文件预览
- 分隔符识别
- 列名定位
- 列名提取

真正的数据读取交由 `load_data.py` 完成。

这样可以保证每个模块职责单一，后续维护更加方便。
